Creating the template design for announcing new web trading features

Over the years, Questrade’s web trading platform has expanded to give users many powerful new features which some users are not aware of, even to this day.

With no way to share new features released on the web platform, it was time Questrade worked on a means to communicate new features to web trading users.




Organization

Questrade

Timeline

July-Aug 2022

Design team

1 UX designer
1 visual designer
1 content designer

Skills

Interaction design
Visual & content design thinking
Product & engineering collaboration





The problem

Wait, a new feature was released?

To this day, there are some previously released features that some users are not aware of, which was emphasized in reports done by the user research team.


Questrade needs a way to communicate new feature releases to the web trading platform, so that users can fully maximize their workflow and use of product enhancements.






The challenge

One template, for infinite applications

Creating a pattern out of the feature announcement was an important aspect to the project. Not only would following the same design template be a time-saver for other designers, more importantly it creates consistency for users to recognize a feature announcement immediately.


How might we build a flexible and versatile way to share information about new web trading features?







The process

01: Gathering context on current Questmobile feature announcements

Despite having the pattern already defined on the mobile app counterpart, based on user statistics it is known that lot of web users aren’t on mobile. This means majority of web users haven’t seen the mobile feature announcement pattern at all.


Regardless, it was important to keep some consistency, for the users using both mobile and web.




02: Establishing design goals

Besides exploring and using the web trading platform myself, I also familarized myself with the platform and the users through conversations with real web trading users. Discussing the user journey and personas created by other designers also greatly aided to empathizing with users.


Using this new knowledge, I revisited the project business and design requirements and reflected on the design goals for designing new feature announcements.





03: Competitive analysis on web feature announcements

Upon researching several web platforms and web apps, 4 of the more common patterns for feature announcements were noted.





04: Narrowing the design scope

With there being big unknowns as to what features could be announced using the solution, it was helpful to broadly define and categorize possible use cases.


Upon collaboration with the product team, it was then determined it was safe to assume that small features and feature enhancements would not need to be announced.





05: Exploring solutions

The problem was broken down into 2 aspects to ideate upon.


Part 1: Feature explaination

Displaying media and relevant information in a simple educational manner.
How might we intuitively explain how the feature works?

Part 2: Feature visibility

Entry point of announcement, visability of new features.
How might we tell users a new feature was released?
How might we tell the users where the new feature is located?


14 different solution concepts were explored (yay!!). The competitive analysis was used as a starting point to generate ideas.


Keeping in mind that the design would be a template, a recently released feature was used as a placeholder feature. However, it was essential to think broadly regarding the placement and location of where potential new features may live.




06: Assessing the top explorations

Part 1: Focus on feature explaination


Concept #1: Guided tour

Visibility: Very noticable.
Educational impact: Very educational, teaches users all about the features.
Minimal disruption: Highly disruptive.
Versatility: Difficulty being used as a template, would need to define extensive rules for placement/location.

Concept #2: Fixed raised side panel

Visibility: Users will immediately see modal at the corner of the page.
Educational impact: Sufficient but missing connection to actual feature on page.
Minimal disruption: May block a bit of content on smaller screen, however this promotes sense of urgency to view announcement.
Versatility: Can be used for all new features.

Concept #3: Full screen popup

Visibility: Very noticable.
Educational impact: Somewhat educational, popup prevents user from seeing the new feature on the page.
Minimal disruption: Highly disruptive.
Versatility: Applicable for all types of new features.

Concept #8: Top banner with no media & no carousel

Feedback from content designer: Carousel is preferred as it is hard to digest multiple announcements at once. Should avoid announcing 2 features through content, and break up the feature announcements instead.

Concept #9: Inline banner with content

Visibility: Varying levels of discoverability for users with more panels on the right side.
Educational impact: Somewhat educational.
Minimal disruption: Low disruption.
Versatility: How would this work for features on other pages, where there might not be blank space to put an inline banner?

Concept #14: Hybrid concept of guided tour (concept #1) and current education mode panel behaviour

Iteration #1: Popup constrained to both side margins

Visibility: Depending on location of new feature on the page, announcement modal may or may not be discoverable. For elements housed at the bottom of the page, the announcement modal may be hidden on first glance.
Educational impact: Somewhat educational.
Minimal disruption: Right side interferes with support menus, informational notifications like feature announcements do not belong on the support side.
Versatility: What would be the location for generic features, like dark mode, which is not housed on the page?



Iteration #2: Popup constrained to left margin only

Visibility: Dependent on location of new feature on page.
Educational impact: Keeping the popup on the left margin will solve the issue of preventing interference with the support menus, however there is a lack of correlation between announcement modal and feature being announced.
Minimal disruption: Will interfere with education mode panel, when education mode is turned on.
Versatility: Would need to define rules for generic features.



Part 2: Focus on new feature visibility


Concept #4: Entry point in help section

Visibility: Easy to notice a notification in the help section, but whether users are inclined to go through the flow to open the notice is dependent on the user.
Educational impact: Very little educational impact in showing how a new feature may work. Lack of correlation between actual announcement and feature, especially with steps inbetween before seeing the announcement.
Minimal disruption: Very little disruption to current flow.
Versatility: Would be able to apply to almost all new features.

Concept #13: NEW tag inline with new feature

Visibility: For inline new features, very discoverable!
Educational impact: Would need to be accompanied by informational modal.
Minimal disruption: May disrupt visual design spacing in relation to other elements.
Versatility: Difficulty being used as a template solution, would need to define extensive rules for placement/location.





07: Making design decisions

After evaluating all concepts, concept #2 was selected as the final solution due to the strong versatility, minimal disruption, and high visability. However, the educational impact could be improved and was an aspect to be explored further.


How might we strengthen the connection between the feature announcement, with the feature location and how it works?


Design decision #1: Using descriptive animations to strengthen visual correlation

While static images are effective to convey what a feature is, animations can deepen visual understanding and allow more complex ideas to be communicated quickly and effectively.
The goal of using animations is to demonstrate the concept of a new feature by showing how it works, which may improve learning.

Design decision #2: Announcement only shown on the relevant page

For the appearance logic of the announcement, the modal is only to appear on the same page the new feature is on. This will reduce the distance between the announcement and the new feature being announced.

Design decision #3: Additional button for when there is only 1 announcement

For >1 new features, it would follow the same carousel style of grouping announcements.


For 1 new feature, it was decided to add another CTA which could be used for many applications. The default application is to dismiss the announcement. This means there are 2 ways to close the modal, which each serves a different user journey; one for users who immediately want to close it, and one for users who read the entire announcement. Being a template, this button can encourage use of the new feature by redirecting the user to settings or enabling the new feature right on the announcement.





08: The final solution

There were still questions that needed to be answered from the collaboration with visual and content designers (ex. the grid for raised elements), but also still more UX decisions that needed to be made regarding feature behaviour. This includes appearance logic, dismissal logic, maximum number of announcements, interaction with other features, etc. Edge cases also needed to be addressed, for which developer collaboration was cruicial. For example, how would the announcement look when the carousel reaches the last item?

Learnings

01: Stay high level!!

With template design, it’s extremely important to stay broad and take versitility highly into account when making decisions. It’s hard to train your brain to think broad (especially when there are many unknowns as to the type of new features that can be announced), but working in a lower fidelity and focusing purely on concept can be helpful.

02: Cover all the edge cases

For >1 new features, it would follow the same carousel style of grouping announcements.


Working with content and visual designers throughout my internship really emphasized my focus on designing for edge cases. It’s easy to get caught up in the details of visual behaviour or the type of content used, but really important to think about how the new feature impacts the product as a whole, and interacts with the current features. Especially with template design, thinking needs to be adaptive and conceptual.
