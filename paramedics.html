Mass casualty patient triaging platform

Working on a platform for the Region of Waterloo Paramedic Services (ROWPS) to help triage and track patients during mass casualty events







Context

Since fall 2020, I’ve been volunteering my time on UW Blueprint, a club aiming to make technology accessible to community non-profits without the funding. Each term I’ve got to work on a project team with other students, and each project team works with a non-profit with a need to help build them a solution and equip them with the tools they need.

My first term on UW Blueprint in fall 2020, I was placed on a project that was building the Region of Waterloo Paramedic Services (ROWPS) a platform to better manage patient triage and track patients + event information in real time. It was essentially a structured system for knowledge management that would be able to be viewed, maintained, and updated by all relevant users.


Project Details

TEAM: 1 product designer (me!) + 1 product manager + 1 project lead + 8 developers
TIMEFRAME: Sept-Dec 2020
MY ROLE: Interaction & visual design, iterative design, product thinking







Wrapping up the project

I joined the project 8 months in, and my focus was on polishing up existing functionalities, and improving + redesigning features based on usability testing + new technical capabilities.








Getting onboarded

What is this platform being built?

A platform allowing the ROWPS to better triage patients, track locations, and obtain patient status while providing paramedic services during a mass-casualty event.



Where did I pick up from? What still needed to be done?

In terms of design, the design system had been created and the main hi-fidelity designs and flows were designed out. However, several key features and functionalites had yet to be designed--which is where I came in!

Besides the redesigns I mention further down, some of the functionalities that still needed to be completed included...

  • Delete & restore patient
  • Maps
  • Offline handling
  • Real time updates
  • Error & edge cases
  • Login, unauth, & branding

While I’m not going to go into the details of design decisions made for these functionalities, feel free to connect to hear more! For this case study, I really want to focus on the major design changes & decisions made to the existing application as a result of usability testing.









Major feature changes

Testing, Testing, 123...


Usability testing

Usability testing with the ROWPS was a super cool experience! My blueprint team got on a google meets call with them and ran through common scenarios a user would need to do. The twist? We were able to witness them testing it in a simulated paramedics mass casualty event scenario! The paramedics had even set up stations around the room, complete with fake patients they would be helping, and several of them using the platform together in different “collision points”.



Takeaway Action Item
Yellow status was straining on their eyes to read and hard in the field Changed color of text to black and kept color focused on side ribbon
Maps wasn’t an intuitive feature and wasn’t well integrated with other aspects of the application Learned more about what their team would need from maps feature for the redesign
Appreciated the way deleted patients were handled (greyed out patients at the bottom)
Had trouble transporting patient from on scene to hospital Changed the approach to adding resources on the fly in the redesign
Overall was able to navigate through and complete tasks well






With the key insights gained from user testing as well as new information of technical capabilities from our developers, our team re-evaluated several sections of the platform.








Feature #1

Transporting a Patient


Old flow

This was the current flow for the scenario of when a paramedic would need to mark down that a patient was being transported to a hospital. The flow would require the user to select hospital and ambulance numbers.




Improving the user flow

As shown above and mentioned in user testing, there were many aspects of the flow that needed revamping.


How might we make better use of the existing resource management functionality so that we can more accurately simulate the transport patient to hospital experience?



User flow iteration #1: Transport information

For this first iteration, after the user clicks “CONTINUE TO TRANSPORT” (the transportation confirmation), the page would expand and would auto scroll down to the new section where the user could select transportation details.

  • This iteration definately fit the feel of the form and matched the rest of the flow, without being too abrupt!
  • It was nice having the transportation information on the same screen, because of easy access to previous information
  • The auto scroll interaction is not as optimal for iPad

User flow iteration #2: Transport information

The use of two pages was then tried. After the user clicks “CONTINUE TO TRANSPORT”, a new page would be opened with the transport information, with the ability to navigate back and forth between pages to edit patient information.




User flow iteration #3: Add new hospital to transport list

The flow was then revamped by changing the add resource part to the flow. Many different ways were explored, as shown below. All of these explorations would open a mage page that allows the user to edit the existing resources and make them active/inactive to edit the resources shown in the multi-select list, and another option existed to completely add a new resource.




Okay okay let’s cut to the chase... what was the final flow that was chosen and why?




User flow iteration #4: Final iteration

The final iteration moved away from giving full access to resource management, but just allowed users to select from a dropdown of the hospitals that were active, or to add a new one if they really needed to. Adding a new resource is the last option on the list, as we want to discourage adding a new resource on the fly.




Final flow


Let’s see it in action

Overall, improved the experience and flow of transporting a patient from on scene to in transport to hospital within the app.
Instead of having only one screen and not being able to edit/view transport information once submitted, this two tab approach allows users to be able to navigate easily between forms.





Feature #2

Add New Event & Collision Point


Old flow

The existing user flow was as follows below. However as mentioned above, from usability testing it was concluded that the use of maps functionality within the entire platform could be improved by incorporating maps better within the creation of events and collision points. This is because it would allow the user to add pins to certain locations right during creation, instead of having to exit and take the extra step to navigate or remember to navigate to the maps section of the application to add a pin.
At this point of the project, the product team also got new information from the developers about technical feisability and limiations.





How might we enable users to have a better understanding of marked locations through creating a more effective and efficient mapping experience?





Iteration #1

As it was important to attempt to repurpose as much of the existing platform as possible (for the developers but also for the user experience, as it is helpful to keep pre-existing and familiar patterns), the first iteration tried repurposing the old Add New Event screen.




Iteration #2

As we liked the live map experience on the right, another approach taken was to then refactor the first page, Add New Event, to fit the map experience as well. To keep the user from being overwhelmed, we broke the first page into general information and then got the user to take another step in adding a new event.

Reflecting on my experience


Key Learnings
  • Work with the right screen sizes!!! The original screens used iPad Pro frames, but it turns out the paramedics would be using a much smaller iPad screen size. As the developers really needed pixel measurements for implementation, I had to figure out work arounds, and spend time resizing key screens
  • Designing on a smaller iPad screen size! The keyboard takes up around of the screen real estate when the iPad is used horizontally (which was the case here), and that definately presented challenges. It’s important to always consider interactions, and not just visual design
  • Usability testing with the actual paramedics who would be using the app! It was so cool to see how the different users approached using the app, and was a great experience in iterating on the designs and continually making improvements after usability testing
  • Error handling, offline, real time updates, which is especially important for an app like this
  • Balance between good visual design and including all the information a user would need
